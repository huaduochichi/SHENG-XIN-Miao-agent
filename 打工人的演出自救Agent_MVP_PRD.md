# 打工人的演出自救 Agent：一个帮忙守住生活热爱的票务情报助理

## MVP PRD

版本：v0.2  
日期：2026-05-15  
阶段：比赛 MVP / 可演示版本  
产品形态：Web App + 移动端适配 + 飞书提醒集成

---

## 1. 项目背景

演出爱好者经常面临信息分散、开票节奏快、预约和抢票节点容易错过的问题。尤其是工作、学业压力较大的用户，很难持续关注大麦、猫眼、秀动、剧场官网、艺人公告、音乐节官方渠道等多个来源，导致很多感兴趣的演出在发现时已经售罄或错过预约。

本项目希望通过 Agent 自动完成演出情报搜索、信息抽取、结构化治理、可信度提示和提醒创建，让用户从“反复刷信息”转变为“确认重要情报并安排待办”。

项目边界明确：本产品不做自动抢票，不绕过票务平台机制，只做公开信息检索、整理、提醒和用户确认后的待办创建。

---

## 2. 产品定位

产品名称：

**打工人的演出自救 Agent：一个帮忙守住生活热爱的票务情报助理**

一句话描述：

面向演出爱好者的 AI 票务情报助理，根据用户偏好自动搜索演唱会、音乐节、脱口秀、话剧、音乐剧等演出信息，整理关键开票节点，并在用户确认后创建飞书待办或日历提醒。

核心价值：

- 帮用户减少重复检索成本。
- 帮用户更早发现感兴趣的演出。
- 帮用户明确预约、开票、候补等关键行动节点。
- 保留用户最终决策权，避免 AI 过度替代判断。

## 3. MVP 目标

### 3.1 产品目标

MVP 需要完成一个清晰闭环：

```text
用户设置偏好
→ Agent 自动执行 Web Search
→ 抽取并整理演出情报
→ 展示匹配原因、可信度和不确定点
→ 用户确认是否关注
→ 创建飞书待办 / 飞书日历提醒
→ 记录 Agent 运行日志与效果指标
```

### 3.2 比赛目标

MVP 需要支撑比赛评分中的重点能力展示：

- 技术深度：体现多 Agent / Skill 编排、信息抽取、去重、可信度判断、可解释参考。
- 工程完整度：覆盖采集、治理、模型调用、后端接口、前端交互、提醒集成、运行日志。
- 业务价值：解决真实的“错过演出开票”问题，并具备个人、校园社群、企业员工福利等扩展场景。
- 合规意识：只处理公开信息，用户授权后才创建提醒，不执行交易行为。

## 5. MVP 范围

### 5.1 本期必须实现

1. 用户偏好设置
   - 常驻城市。
   - 可接受周边城市。
   - 喜欢的艺人、乐队、演员、剧团或关键词。
   - 感兴趣的演出类型。
   - 预算区间。
   - 可接受时间偏好。

2. Web Search Skill
   - 根据用户偏好自动生成搜索 query。
   - 搜索公开网页中的演出信息。
   - 保留搜索关键词、来源链接、抓取时间。

3. 演出信息抽取
   - 从搜索结果和页面摘要中抽取结构化字段。
   - 字段包括演出名称、类型、城市、场馆、演出时间、预约时间、开票时间、票价、平台、来源链接、状态。

4. 情报治理
   - 对同一演出进行基础去重。
   - 标记字段完整度。
   - 标记来源可信度。
   - 标记信息不确定点。

5. 参考判断
   - 系统展示“推荐关注 / 需要确认 / 可能不适合”等参考标签。
   - 展示原因，例如城市匹配、艺人匹配、预算可能符合、开票时间缺失。
   - 最终是否创建提醒由用户决定。

6. 用户确认机制
   - 用户可以对每条情报执行：
     - 关注
     - 忽略
     - 创建抢票提醒
     - 加入候补观察
     - 稍后再看

7. 飞书提醒集成
   - 用户授权或配置飞书提醒方式。
   - 创建飞书待办。
   - 创建飞书日历事件。
   - 记录提醒创建成功或失败状态。

8. Agent 运行日志
   - 展示本次搜索了哪些 query。
   - 展示发现了多少条候选情报。
   - 展示抽取成功率、去重数量、提醒创建数量。
   - 展示失败原因，例如页面不可访问、字段缺失、飞书接口失败。

### 5.2 本期不做

- 不做自动抢票。
- 不做票务平台登录态爬取。
- 不做绕过验证码、排队或风控。
- 不做原生 iOS / Android App。
- 不做邮件提醒。
- 不做复杂多用户权限体系。
- 不做全平台全城市覆盖。

核心模块：

- 今日新发现。
- 高匹配情报。
- 即将开票。
- 信息待确认。
- 来源可信度。

演出卡片字段：

- 演出名称。
- 类型标签。
- 城市 / 场馆。
- 演出时间。
- 预约 / 开票时间。
- 票价范围。
- 来源平台。
- 可信度。
- 匹配原因。
- 不确定信息。
- 操作按钮：关注、忽略、创建提醒。

### 6.2 页面二：待确认情报

定位：承接 Human-in-the-loop，突出用户最终决策权。

核心模块：

- 系统参考标签。
- 匹配原因解释。
- 风险或缺失字段提示。
- 用户确认操作。

示例：

```text
系统参考：推荐关注
原因：
- 命中你关注的乐队“草东没有派对”
- 城市为上海，符合常驻城市
- 演出时间为周末
待确认：
- 暂未找到准确开票时间
```

### 6.3 页面三：我的抢票待办

定位：展示用户已确认关注并创建提醒的演出。

核心模块：

- 今日待办。
- 未来 7 天开票。
- 候补观察。
- 演出前提醒。
- 飞书同步状态。

待办状态：

- 未开始。
- 即将开始。
- 已同步飞书。
- 同步失败。
- 已完成。
- 已取消。

### 6.4 页面四：偏好画像

定位：管理用户兴趣输入，作为 Agent 搜索与匹配的依据。

字段：

- 常驻城市。
- 周边城市。
- 关注艺人 / 乐队 / 演员 / 剧团。
- 演出类型。
- 预算上限。
- 时间偏好。
- 是否开启飞书提醒。

### 6.5 页面五：Agent 运行日志

定位：面向比赛演示，证明系统不是静态 Demo。

展示内容：

- 最近一次运行时间。
- 生成的搜索 query。
- 搜索结果数量。
- 有效情报数量。
- 去重数量。
- 字段完整率。
- 用户确认率。
- 飞书提醒创建成功率。
- 错误日志。

### 6.6 当前前端 Demo 设计思路与改造计划

本轮前端 Demo 的目标不是做一个只展示静态信息的看板，而是让评委和用户能够完整体验“先表达偏好，再由 Agent 发现情报，最后由用户确认行动”的产品闭环。

#### 6.6.1 首次使用体验

首次进入产品时，页面采用左右分栏：

- 左侧为品牌引导区：展示产品名、产品 Logo、主题萌宠和一句核心价值说明，建立“有人帮我守着演出动态”的陪伴感。
- 右侧为偏好筛选面板：让用户快速设置城市、演出类型、喜欢的艺人/剧目、预算和可观演日期。
- 用户可以选择“保存偏好并启动雷达”，也可以选择“暂时跳过”，先用默认偏好进入 Demo。
- 首次偏好筛选不是全屏设置页，也不是遮罩弹窗；它应是首屏中央的左右分栏面板：左侧固定品牌与萌宠引导，右侧为可滚动的偏好筛选面板。
- 用户跳过后，系统必须记录“默认偏好待完善”状态，并在后续主界面提供显眼入口让用户随时修改偏好。
- 首次设置完成后进入演出雷达主界面，后续可以随时从侧边栏、移动端底部导航或默认偏好提示条重新进入偏好设置。

#### 6.6.2 主界面信息架构

主界面保持工作台结构，避免营销页式表达，核心信息围绕“情报处理效率”组织：

- 左侧导航：产品标识、萌宠状态提示、演出雷达、待确认、抢票待办、运行日志、偏好设置；当用户跳过首次设置时，入口文案调整为“完善偏好”。
- 顶部状态栏：展示情报总数、推荐关注数、待处理数、当前城市范围和最近一次 Agent 运行状态。
- 内容区：根据当前任务切换为卡片流、确认列表、提醒清单或日志流水；如果用户跳过首次设置，内容区顶部展示“当前使用默认偏好”的提示条和“完善偏好”按钮。
- 移动端：底部导航覆盖雷达、待确认、待办、偏好和日志，保证关键页面可达，即使用户跳过首次筛选也能再次进入偏好面板。

#### 6.6.3 本轮必须补齐的可用功能

- 偏好设置可保存、可跳过、可返回，并能触发一次模拟 Agent 运行。
- 演出雷达支持推荐关注、需要确认、即将开票、已关注和全部筛选。
- 演出卡片支持关注、创建提醒，点击后打开详情抽屉。
- 待确认页支持按推荐、缺失开票时间、来源待核验等维度筛选。
- 详情抽屉展示来源、可信度相关字段、匹配原因、风险提示和用户决策。
- 抢票待办页展示今日、未来 7 天、候补观察，并支持取消已创建提醒。
- Agent 运行日志展示 Query、检索、抽取、去重、写入等步骤，证明 Demo 不是纯静态页面。
- 用户操作后出现即时反馈，减少“按钮点了没反应”的不可用感。

#### 6.6.4 视觉方向

- 整体风格：深色工作台 + 高对比信息层级，突出情报、风险和行动按钮。
- 品牌表达：Logo 与主题萌宠只在 onboarding 和侧边栏中承担引导角色，不挤占主任务区域。
- 卡片设计：演出图片作为识别线索，卡片中优先展示时间、城市、开票、票价和来源。
- 情绪设计：萌宠负责降低工具焦虑，但不替代关键业务信息，避免把产品做成纯装饰型页面。
- 可访问性：所有关键操作按钮需要有明确文案，移动端必须能完成偏好设置、查看详情和创建提醒。

#### 6.6.5 Demo 验收路径

比赛演示时推荐按以下顺序展示：

1. 首次进入，展示左侧 Logo + 萌宠引导，右侧偏好筛选。
2. 保存偏好并启动雷达，展示运行中状态。
3. 在演出雷达中切换“推荐关注 / 需要确认 / 即将开票”等筛选。
4. 打开一条演出详情，解释匹配原因、风险提示和来源。
5. 点击创建提醒，展示即时反馈和抢票待办页新增状态。
6. 打开运行日志，展示 Query、抽取、去重和提醒创建指标。

## 7. Agent 工作流

### 7.1 Query Planner

输入：

- 用户偏好。
- 当前日期。
- 城市和演出类型。

输出：

- 一组搜索 query。

示例：

```text
上海 音乐剧 开票 2026
五月天 上海 演唱会 开票
草东没有派对 巡演 2026 票
上海 脱口秀 专场 开票
杭州 音乐节 早鸟票 2026
```

### 7.2 Search Agent

职责：

- 调用 Web Search API。
- 获取标题、摘要、URL、发布时间等信息。
- 过滤明显无关或过期结果。

### 7.3 Extractor Agent

职责：

- 从搜索结果或页面正文中抽取演出结构化信息。
- 输出统一 JSON。

目标 JSON：

```json
{
  "event_name": "草东没有派对 2026 巡演上海站",
  "category": "演唱会/Livehouse",
  "city": "上海",
  "venue": "待确认",
  "event_time": "2026-06-20 19:30",
  "reservation_time": null,
  "ticket_sale_time": "2026-05-18 12:00",
  "price_range": "380-880",
  "ticket_platform": "大麦",
  "source_url": "https://example.com",
  "status": "即将开票",
  "uncertain_fields": ["venue"],
  "extracted_at": "2026-05-11T10:00:00+08:00"
}
```

### 7.4 Verifier Agent

职责：

- 合并重复情报。
- 计算字段完整度。
- 判断来源可信度。
- 生成信息不确定点。

可信度参考：

- 官方渠道、票务平台：高。
- 主流媒体、剧场公告：中高。
- 社交平台转述：中。
- 无明确来源的聚合内容：低。

### 7.5 Preference Explainer

职责：

- 给出参考标签。
- 说明匹配原因和不确定点。
- 不直接替用户创建提醒。

参考标签：

- 推荐关注。
- 需要确认。
- 可能不适合。

### 7.6 Reminder Agent

触发条件：

- 用户点击创建提醒。

职责：

- 根据开票时间、预约时间、演出时间生成待办。
- 调用飞书任务或飞书日历接口。
- 记录同步状态。

## 11. 合规与安全

合规原则：

- 只检索公开网页信息。
- 不采集票务平台登录后数据。
- 不绕过验证码、排队、风控或反爬机制。
- 不做自动购票或自动抢票。
- 所有来源链接保留，便于用户自行核验。
- 用户授权后才创建飞书任务或日历。
- 用户偏好数据仅用于演出情报匹配。
- 用户可删除偏好、决策记录和提醒记录。

比赛材料中需要明确说明：

> 本项目定位为信息发现与提醒工具，不参与票务交易，也不替代用户完成购票行为。

## 12. 技术方案

### 12.1 推荐技术栈

- 前端：Next.js + TypeScript + Tailwind CSS
- 后端：Next.js API Routes
- 数据库：Supabase PostgreSQL
- AI 模型：OpenAI API 或兼容模型 API
- Web Search：Tavily / SerpAPI / Bing Search API
- 定时任务：Vercel Cron
- 飞书集成：飞书开放平台任务 API / 日历 API
- 部署：Vercel + Supabase

### 12.2 架构原则

- MVP 采用单体全栈 Web App，降低部署复杂度。
- Agent 编排先用明确的 workflow 函数实现，后续可升级到 LangGraph。
- 搜索、抽取、校验、提醒各模块保持独立，便于扩展。
- 所有 Agent 运行过程写入日志表，便于可观测和比赛演示。

---

## 13. MVP 验收标准

### 13.1 功能验收

- 用户可以完成偏好设置。
- 系统可以根据偏好自动生成搜索 query。
- 系统可以返回至少 10 条候选演出情报。
- 系统可以抽取并展示核心字段。
- 系统可以标记可信度、字段缺失和匹配原因。
- 用户可以对情报做关注、忽略、创建提醒操作。
- 系统可以成功创建至少一种飞书提醒。
- Agent 运行日志页面可以展示搜索、抽取、去重、提醒状态。

### 13.2 演示验收

比赛演示时需要完成以下路径：

1. 展示用户偏好。
2. 点击“运行演出雷达”。
3. 展示 Agent 自动生成的搜索 query。
4. 展示新发现的演出情报卡片。
5. 解释可信度和匹配原因。
6. 用户点击“创建抢票提醒”。
7. 展示飞书待办或日历创建成功。
8. 展示 Agent 运行日志和关键指标。

## 14. 后续迭代方向

MVP 之后可以扩展：

- 支持更多城市和演出类型。
- 支持多用户账号体系。
- 支持飞书消息机器人主动推送。
- 支持团队/社群共享演出清单。
- 支持票务状态变化监控，例如二开票、候补、售罄恢复。
- 支持 PWA 添加到手机桌面。
- 支持从图片海报中 OCR 抽取开票信息。
- 支持基于用户反馈优化搜索 query 和匹配解释。

---

## 15. 比赛答辩亮点

本 MVP 可以在答辩中强调：

- 不是简单爬虫，而是从 Web Search 到结构化治理再到飞书提醒的完整 AI 工程链路。
- 不是 AI 替用户做决定，而是 Human-in-the-loop 的可解释情报助理。
- 不是静态 Demo，而是有运行日志、质量指标和外部系统集成的可演示产品。
- 业务上可以从个人用户扩展到校园社群、企业员工福利和城市文娱运营场景。
- 合规边界清晰，不参与票务交易，不绕过平台规则。
